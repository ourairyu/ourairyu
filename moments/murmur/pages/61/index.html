今年 Q1 的人生 OKR（从现在起到三月末）——

Objective：验证并跑通我的 UI 组件体系。

Key Results：

  1. 半成品 UI 组件集合 Petals 完成 90% 以上；
  2. 基于 Petals 写组 Vue 的结构组件；
  3. 两套风格配置，分别是 Ant Design 和 Element。

Q1 末或 Q2 初来个复盘总结。

前端 UI 开发这么混乱的原因，一是前端本身的原因，即抽象程度、方法论和基础设施等；二是 HTTP API 定义混乱。

传统的接口方式支持不了复杂的模式化，所以即使没有 GraphQL 也会出现类似的东西。

等表现层的鸡零狗碎弄好了，就可以潜入逻辑之海去探索真理了。

人生 OKR——

首先要有人生目标，将其拆成十年目标、五年目标、年目标和季度目标；每级目标分解出几个关键结果，除了季度的关键结果会成为下一级目标的参考，季度的关键结果会成为更细分的任务落实到每一天。

人生 OKR 与工作 OKR/KPI 最好是重叠的，这样可以事半功倍，更好地提高完成度和质量。

每日之事都要记录，以周或月的频次复盘。

个人效率——

人脑是单线程的，至少我是，无法同时处理多件事情，尤其是写作、编程、思考等需要大段连续时间的事情，每次打断都需要用很多的时间重新找状态。做这类事情时最好「与世隔绝」，尽可能减少外部干扰。能够在心流状态完成一件事，无疑是最高效的。

提高个人效率的另一方面，就是「压缩时间」：一是利用好「空闲时间」，如在拉屎、洗澡、走路、开车等不太需要大脑的事情上进行思考；二是减少时间消耗，如拉屎、洗澡、走路、吃饭等事速度快点。

其他——

任何重复的且不需要人决策的工作/事情都可被自动化。

昨天在群里讨论到一种现象——人们总是只知道自己所知道的，对已经存在的自己不知道的部分一无所知，然后总会「重新发现」。

这与我之前说的「历史总是相似的」实际上是一回事，可分为四个方面——

  1. 信息的存储方式；
  2. 信息可传递并尽可能在传递过程中减少流失；
  3. 信息检索与接收方式；
  4. 某个人去寻找信息。

前三个是整体需要解决的，最后一个是个体问题。

对于个体来说，看书是增加知识的一种方式，思考是增长智慧的一个途径，但有时需要更高效、更精准地临时找到某个知识，现有的网络环境和手段暂时无法很好地满足这一需求。

如果 Web 存储了所有有效知识并且可访问（不会失效，没有管制等），人的大脑直接与 Web 相连，某个装置将意识直接转化为精确的想要搜索的知识描述，搜索结果也能直接转化为让大脑接收、理解的东西，这无疑是种非常高效的方式。

这会涉及到本体工程、语义网、AI、脑机接口、神经科学等等，我觉得是个值得探索的方向。

历史总是相似的，为什么前人犯过的错，后人不吸取教训再去犯—— 前人的事件、知识没有记录； 前人的事件、知识有记录； 记录遗失； 记录顺利流传下来； 后人在解读时意思与前人所表达的不符； 因为前人的记录总和可以看作是无限，后人无法全部了解并吸收； ...查看详情

光一个 type prop 我就纠结半天——

这个 prop 的语义到底是什么？是风格类型？还是功能类型？或是其他？
